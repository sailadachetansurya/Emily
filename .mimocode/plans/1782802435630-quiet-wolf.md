# Plan: Internal Reasoning/Self-Critique Loop for Emily Pipeline

## Goal

Add an optional internal reasoning loop that has the LLM reason about its response approach *before* generating, then optionally critique the output *after* generating — reducing the safety processor's post-hoc intervention rate from ~30-40% to ~5-10% on high-risk requests.

## Architecture Summary

The reasoning loop is a new stage (`stages/reasoning_loop/`) that wraps the LLM generation call. When activated (emotional risk above threshold), it runs:

1. **Reasoning step**: LLM produces a brief internal trace (chosen approach, rationale, risk assessment)
2. **Enhanced generation**: The trace is injected into the system prompt, guiding the LLM toward compliant output
3. **Critique step** (optional): LLM evaluates its own output against policy rules
4. **Regeneration** (if critique fails): Tighten constraints and regenerate, up to N times

When inactive (low risk or disabled), the pipeline runs unchanged — zero overhead.

### `src/pipeline/stages/reasoning_loop/prompts.py`
Pure functions returning prompt strings:
- `build_reasoning_prompt(policy, emotion, memory, user_input)` — asks LLM to choose approach + rationale
- `build_critique_prompt(policy, response_text, rationale)` — asks LLM to evaluate compliance
- `build_enhanced_system_prompt(original, trace)` — injects reasoning guidance into system prompt
- `build_tightened_prompt(bundle, critique, trace)` — adds DO NOT constraints from critique violations

### `src/pipeline/stages/reasoning_loop/critic.py`
`PolicyCritiqueEvaluator` — calls LLM with critique prompt, parses JSON output. Fail-open on parse errors.

### `src/pipeline/stasoning_loop/orchestrator.py`
`ReasoningLoopOrchestrator` — the main stage class:
- `should_activate(score_breakdown: dict) -> bool` — checks emotional_risk >= threshold
- `process(prompt, policy, score_breakdown) -> (GenerationResult, list[IterationRecord])` — full loop

## Config Fields

| Field | Type | Default | Range | Purpose |
|---|---|---|---|---|
| `reasoning_loop_enabled` | bool | `false` | — | Master switch |
| `reasoning_loop_max_iterations` | int | `2` | 1–5 | Max critique-regen cycles (1 = reason + generate, no critique) |
| `reasoning_loop_activation_threshold` | float | `0.5` | 0.0–1.0 | emotional_risk score to trigger loop |

## Key Design Decisions

1. **Fail-open critic**: If the LLM's critique output can't be parsed as JSON, return `CritiqueResult(compliant=True)`. Never block production responses due to formatting issues.
2. **Stage-internal models**: `ReasoningTrace`, `CritiqueResult`, `IterationRecord` live in the stage, not in `contracts/models.py`. The orchestrator only sees `GenerationResult` (with iteration data in metadata).
3. **No cross-stage imports**: The stage accepts `score_breakdown: dict` and `ResponsePolicy`, not `PolicyDecision` — avoids importing from `policy_engine`.
4. **`was_regenerated` unchanged**: Reasoning loop regeneration is tracked in telemetry traces, not in the `SafeResponse.was_regenerated` field (which is for safety processor regeneration).
5. **Config defaults to off**: Zero impact on existing behavior until explicitly enabled.

## Test Strategy

All tests mock `LocalLLMClient.generate()` — no real Ollama calls.

- `test_should_activate_above_threshold` / `test_should_activate_disabled` / `test_should_activate_below_threshold`
- `test_parse_reasoning_valid_json` / `test_parse_reasoning_invalid_json` (fail-open)
- `test_parse_critique_valid` / `test_parse_critique_invalid` (fail-open)
- `test_critique_passes_first_try` — loop exits after one generation
- `test_critique_triggers_regeneration` — loop regenerates, then exits
- `test_max_iterations_respected` — stops even if still non-compliant
- `test_single_iteration_no_critique` — when max_iterations=1
- `test_full_pipeline_with_mock_llm` — end-to-end with mocked LLM
- `test_loop_bypass_when_inactive` — llm_client.generate() called exactly once
- `test_fallback_generation_still_works` — Ollama down scenario
